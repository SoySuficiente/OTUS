Есть внешняя система данных СКУД
И есть ИТ-система управления рисками
И субъект совершает действие, фиксируемое системой СКУД
Тогда СКУД отправляет информацию о действии в топик Kafka
И consumer Scud читает информацию:
•	Id субъекта
•	Id действия
•	Дата и время действия
•	Описание действия
•	Местонахождение субъекта


Есть consumer Scud
И есть микросервис Распределитель событий
И от внешней системы СКУД пришла новая информация о событиях
Тогда consumer Scud передает сообщения в Распределитель событий
И Распределитель событий отбирает сообщения по настроенному параметру и передает их в Сервисы тревог


Есть внешняя система данных Рейсов
И есть ИТ-система управления рисками
И оператор формирует новый рейс для субъекта
Тогда система Рейсов отправляет информацию о рейсе в топик Kafka
И consumer Flight получает информацию:
•	Id рейса
•	Id субъекта
•	Номер машины
•	Дата и время выполнения рейса
•	Контрольные точки следования
•	Тип ввоза/вывоза


Есть consumer Flight
И есть микросервис Справочных данных
И от внешней системы Рейсов пришла информация о рейсе
Тогда consumer Flight передает сообщения в микросервис Справочных данных
И микросервис Справочных данных проверяет информацию о рейсе по id и создает новую запись в свой базе данных
Но если по id рейса ранее поступала информация
Тогда микросервис Справочных данных апдейтит запись в своей базе данных


Есть Сервис тревог
И есть микросервис Справочных данных
И от Распределителя событий пришло сообщение о совершении события субъектом
Тогда Сервис тревог по id субъекта запрашивает информацию о рейсе у микросервиса Справочных данных
И начинает обрабатывать информацию согласно заложенной индивидуальной бизнес-логике


Когда Сервис тревог вычисляет необходимость формирования тревоги
Тогда Jira получает сообщение о создании карточки тревоги на канбан-доске проекта
